数据结构

数值处理

取整

  • /整数除法，截取整数部分
  • math.Ceil 向上取整
  • math.Floor 向下取整
  • math.Round 四舍五入

其他数值处理

math库里面还有三角函数可以使用，这里我就不做示例了。

标准输出

  • Scan：空白字符分割

如果少一个数据的话，Scan这里就会进行阻塞；当输入的数据多了后，这里只取需要的数据即可，后面多输入的等程序中的下一个Scan读取。

  • Scanf：读取输入，按照格式匹配解析。如果解析失败，立即报错，那么就会影响后面的Scanf。

fmt.Scanf(“%s %d\n”, &name, &age) 这里的%会和后面的非空白字符分不清楚，这个除了用空白字符可以分清楚，其他的字符都会被看出是字符串中的字符，这里建议格式字符串中，一律使用空格等空白字符分割。

线性数据结构

抽象概念

  • 线性表：简称表，是一种抽象的数学概念，是一个有限序列可以存放元素，是一个有序的存放元素的容器。

物理实现

  • 顺序表：使用连续的内存单元来存储该序列的所有元素，内存的顺序就是数据的顺序，可以说是已经排好队的一个容器，数组Array就是一个顺序表；它需要开辟连续的内存空间，变量指向顺序表的第一个元素的内存地址，而容器占用的内存可以被称为 容量(cap)，容量指的就是最多可以容纳多少个元素；当前容器的元素个数也被称为 长度(len) ，是用一个元数据来记录这些值的。

从CURD的方面来分析：

  • C(create)
    • 容器元素个数加1
    • 使用append如同排队，在队尾增加元素
    • insert 插队
      • 在队尾插队的话就相当于append，影响不大
      • 中间插队的话，是要占用当前的位置，将之前的当前位置与其后的元素数据要往后移
      • 当要在队首插入时，就会将所有的元素往后移在插入
      • 当数据量大的时候，这个代价就有点大了，因为挪动数据是需要消耗时间的，一旦数据量很大时，消耗的时间也会很长
  • D(delete)
    • 容器的元素个数减1
    • pod队尾移除一个元素，这样影响最小
    • remove
      • 如果是队尾的话，就是相当于pop
      • 如果在中间离队的话，后面的数据统统往前移
      • 如果在队首离队的话，后面的数据统统往前移
  • U(update)
    • 元素个数不变
    • 首先定位需要修改的元素
      • 使用索引定位，通用公式：”首地址 + 该类型所占的字节数 * 偏移”，所以定位要使用索引来计算得到的元素内存地址，这样的话就不用去都遍历一遍了，效率极高
      • 如果使用内容定位，需要进行内容比较，再通过遍历的方式去挨个比较，这样的话效率低下，要知道最惨的结果就是遍历完了，发现没有对应的这个元素
    • 更新内容
  • R(read)
    • 定位元素
      • 使用索引定位，通用公式：首地址 + 该类型所占的字节数 * 偏移，所以定位要使用索引计算得到的元素内存地址，这样的话就不用去都遍历一遍了，效率极高
      • 如果使用内容定位，需要进行内容比较，再通过遍历的方式去挨个比较，这样的话效率低下，最惨的结果就是遍历完了，发现没有对应的这个元素
      • 使用索引计算与规模n有关，使用固定的步骤就可以计算数元素的位置，时间复杂度为O(1)
    • 获取内容：使用索引直接定位该位置，取走内容
    • 遍历：容器的元素，不管有没有顺序，我们都会不重复的将所有元素挨个摸个遍， 从首地址开始，挨个遍历一遍去内容

前提： 需要看数据的规模量大不大，如果数据规模小的话，到时影响不大，一旦数据规模大起来的话，操作起来就很麻烦；这里的顺序表仅仅适合在尾部做增删的操作，但是这又会出现存储元素空间的扩容的问题。

  • 链接表：每一个元素存储在内存中，但是元素不一定是连续的存储在内存中的，一般都是散落在内存的不同位置，前一个元素指向下一个元素（链接），head首部元素，tail尾部元素；链表是一个容器可以存放元素，定义一个变量指向链表中的第一个元素地址；不需要提前开辟内存空间。
    • 单向链表：指的是前一个元素指向下一个元素的链接
    • 双向链表：指的是前一个元素指向下一个元素，而下一个元素也指向上一个元素，可以理解是双向奔赴的意思
    • 列表（list）往往都是链表来实现的，但是Python是例外

从CURD的方面来分析：

  • C(create)
    • 容器的元素个数加1
    • 尾部追加，原来的尾巴将会指向新的尾巴，容器改tail的指向；没有数据的挪动，很快
    • 中间插入，断开原来的链接，分别和新的元素链接，也没有数据的挪动，这样代价也不大
    • 首部插入，将原来的头和新的头互指一下，容器改head的指向，没有数据的挪动，代价也不高
  • D(delete)
    • 容器元素个数会减1
    • 尾部删除，可以用tail定位尾部，当前尾部的前一个元素就成为新的tail，删除旧的tail的元素
    • 中间删除，遍历定位，当前元素删除，前面那个元素跟后面那个元素重新链接即可
    • 首部删除，用head定位首部，当前head的下一个元素成为新的head，删除旧的head元素
  • U(update)
    • 元素的个数不变
    • 定位元素
      • 使用索引定位，由于元素是散落在内存中的，就不可以使用顺序表的公式来定位元素，首先要找到开头，然后在依次的去遍索引，但是取的每一个节点中保存的下一个元素的地址，在使用该地址去定位下一个元素的元素的位置；这样和顺序表比起来，会略慢一些，但是由于都是使用的地址，效率还是不错的。
      • 使用内容定位，这个需要遍历查找，效率低下
    • 找到元素后直接更新内容即可
  • R(read)
    • 定位元素
      • 使用索引定位，由于元素是散落在内存中的，就不可以使用顺序表的公式来定位元素，首先要找到开头，然后在依次的去遍索引，但是取的每一个节点中保存的下一个元素的地址，在使用该地址去定位下一个元素的元素的位置；这样和顺序表比起来，会略慢一些，但是由于都是使用的地址，效率还是不错的。
      • 使用内容定位，这个需要遍历查找，效率低下
    • 找到内容后取走即可

在数据规模量大时，如果增删在中间或者头部的话（也包括尾部），这里更适合使用链接表，这样就不需要扩容，随便在内存中找一个位置存放当前元素即可，在修改指向（即链接地址）。

数组等类型，如同地铁站排好的队伍，有序，可以插队、离队，可以索引。

链表，如同操场上手拉手的小朋友，有序，但排列随意，或者可以想象成一串带线的珠子，随意盘方在桌上。也可以离队、插队，也可以索引。

数组

数组是由顺序表实现的，容器是不可变的，也就是容器中的元素个数是不可以改变的；数组中必须给定长度，以后数组这个容器就不能增删元素了；是支持索引的，是值类型。

定义

长度和容量

  • cap即capacity，指的是容量，表示可以给数组分配的内存空间可以容纳多少个元素
  • len即length，指的是长度，表示目前容器中有多少个元素

由于创建数组是就必须确定好元素的个数，而且后面是不可以修改长度的，所以不需要预留多余的内存空间，这里的len和cap的值对于数组来说是相等的。

索引

在go语言中是不支持负索引，通过[index]来获取改索引位置上的值，索引范围就是[0,长度-1]，使用索引是最快的定位方式了。

修改

遍历

  • 索引遍历
  • for range遍历

内存模型

  • 数组是必须要在编译时就确定好长度，之后就不能修改长度了
  • 数组的首个元素地址也就是数组的地址（类似于C语言）
  • 所有的元素一个接一个顺序存储在内存中
  • 数组内元素的值是可以改变的，但是元素地址是不变的

一个元素在内存中占用多少字节和元素的类型有关，一个元素的存储单元后面一定是第二个元素的存单元，因为数组是顺序表，这个前面的代码中的每个元素间隔8个字节，自己的电脑也是64位的，这样也符合int类型的定义。

如果数据元素是字符串类型呢？

  • 数组的首个元素的地址就是数组的地址
  • 所有元素都是按照顺序存储在内存中的
  • 元素的值可以被修改，但是元素的地址是不变的

上面的输出的结果，每个元素间隔16个字节是为什么呢？这说明了什么？

字符串是字面常量，一旦定义了就不可以改变了，不同的字符串长度不一，当数组采用string为元素时，经过测试发现存储间隔是一样的，都是占用的16字节，这说明了字符串比我们想象的还要复杂一些，而在这16个字节中还存放着一个指针，指向字符串的内存地址。

值类型

  • 数组是值，值是完全复制的，生成了一个副本
  • 用数组给其他的标识符赋值的时候，值也是被复制了一个副本
  • 函数传入参数的时候，这时的值也是被复制了一个副本
  • 函数返回值用数组，也是会复制，生成一个副本
  • 这样的话就会导致创建的副本太多，可能就会大量的浪费内存资源，这时我们就会使用到切片来防止浪费内存的情况出现

切片

  • 容器可变
  • 长度可变
  • 底层是基于数组的，依赖于顺序表，表现的也像一个可变容量和长度的顺序表
  • 引用类型，和值类型是有区别的

定义

内存模型

切片的本质就是对底层数组一个连续片段的引用，这个片段可以是整个底层数组，也可以是由起始和终止索引标识一些项的子集。

追加

append在切片的尾部追加元素，长度加1

当增加元素后，是有可能超过当前的容量的，当超过当前容量的话，就会触发底层数组的扩容。

长度和容量

  • append一定返回一个新的切片，但是本质上说是新的header（标头值）
  • append可以添加若干个元素
    • 增加若干元素时，当前长度 + 新增元素个数 <= cap 的话就不会发生扩容
      • 原切片使用原来的底层数组，返回的新切片也使用这个底层数组
      • 返回的新切片有新的长度
      • 原切片长度不变
    • 如果增加元素时，当前长度 + 新增元素个数 > cap 的话就需要发生扩容了
      • 这时会生成新的底层数组，新生成的切片使用该新数组，将旧元素复制到新的数组，其后在追加元素
      • 原切片底层数组、长度、容量不会改变

扩容策略

https://go.dev/src/runtime/slice.go

在老版本（1.18-）实际上，当扩容后的cap<1024时，扩容翻倍，容量变成之前的2倍；当cap>=1024时，变成之前的1.25倍。

而在新版（1.18+）阈值就变成了256，当扩容后的cap<256时，扩容翻倍，容量变成之前的2倍；当cap>=256时，newcap += (newcap + 3*threshold) /4，计算后就是newcap = newcap + newcap/4 + 192，即在1.25倍后再加192。

扩容是创建新的底层数组，把原内存数据拷贝到新内存空间，然后在新的内存空间上执行元素追加操作。

切片频繁扩容的话成本会非常高，所以尽量早些估算出要使用的大小，一次性给够，建议使用make来定义切片，常用make([]int, 0, 100)。

其实将一个切片给一个标识符赋值时，这是使用值拷贝，但是切片拷贝的是标头值（header）。标头值内的指针也被复制，刚复制完都是指向的同一个底层数组罢了，但是一旦操作切片扩容了，另一个切片增加元素，那么就不能简单的归结为”切片是引用类型，拷贝了地址”这样的简单的话来解释了，这时就需要根据具体的问题来具体的分析了。

Go语言值全都是值传递，整型、数组这样的类型的值是完全复制，slice、map、channel、interface、function这样的引用类型也是值拷贝，不过复制的是标头值（header）。

截取子切片

就是选取当前切片中的一段生成一个新的切片，但是他们都是共同一个底层数组的，但是标头值（header）中的三个属性会变，截取的格式为slice[start:end]，坚持前包后不包的一个原则。

从上面的代码和图可以看出，截取的这个操作都是从同一个底层数组上取的段，所以子切片和原始切片共同一个底层数组。

  • 通过指针可以确定底层数组是从哪里开始共享的
  • 容量是底层数组从偏移的元素到结尾能容纳多少个元素

截取子切片本质上是利用原来的切片的部分元素，是共用一个底层数组的，所以截取子切片操作是没有可能构建一个新的底层数组的，因此也不会导致扩容。

其实数组也是可以切片，会生成新的切片。

切片的总结：

  • 使用slice[start:end]表示截取切片，新切片的长度为end-start，规则就是前包后不包
  • start的缺省是表示从索引0开始，最大的值是不能超过cap(slice)
  • end的缺省是len(slice)，表示取到末尾，包含了最后一个元素，要注意这里的缺省不是cap而是len
    • a1[4:]相当于a1[4:len(a1)]
  • 如果start和end都使用缺省的话，就相当于从头取到尾
  • start和end同时给定值是，这时的start <= end
    • start、end最大都不可以超过容量值
    • 假设当前容量为8，长度为5，有以下情况
      • s1[:]，这样是可以的，共用底层数组，相当于对header的拷贝，也就是指针、长度、容量都是一样的，也就是cap为8，len为5
      • s1[1:]，这样是可以的，共用底层数组，相当于跳过了底层数组中的第一个元素，header中指针偏移1，长度len为4，容量cap为7
      • s1[1:4]，这样也是可以的，共用的底层数组，掐头、去尾，header中的指针偏移1，长度len为3，容量cap为7
      • s1[:8]，这样是可以的，共用的底层数组，end最多写成8（因为后面不包），如果写成s1[:9]的话就不可以了，end是不可以超过容量的，所以改切片长度和容量都为8，这时如果append这个切片的话就会扩容了
      • s1[8:]，这写法是错误的，因为这时的end用的是缺省为当前长度5，这里需要符合start <= end
      • s1[8:8]，这样写是可以的，共用底层数组，但是这个切片的容量和长度都为0了，要注意这个和s1[:8]的区别，这时这个切片要append的话就扩容生成新的底层数组了
      • s1[7:7]，这样写是可以的，共用底层数组，header中的指针偏移7，长度为0，容量为1，这时这个切片还可以append一个元素，再多添加的话就扩容
      • s1[0:0]，这样写是可以的，共用底层数组，这个切片的长度为0，容量为8
  • 新的切片刚产生时，和原序列（数组、切片）开始共用同一个底层数组，但是每一个切片都自己独立的内存空间保存指针、len、cap
  • 一旦一个切片扩容，就和原来共用的一个底层数组的序列分道扬镳，从此陌路
  • 掐头，会导致首地址的变化，从而会引起cap的变化
  • 去尾，会导致len的变化，但是要append的话就要小心了，添加的元素加上当前的元素个数没超过容量的话添加的元素会覆盖之前的元素，当超过了容量时就要扩容了，和之前的底层数组就没有关系了
