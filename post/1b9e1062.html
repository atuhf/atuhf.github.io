协程

Go协程是与其他函数或方法并行运行的函数或方法，你可以认为它类似于轻量级的线程，正如一个进程里面可以拥有多个线程，而一个线程里面又可以包含多个协程。协程不是被操作系统内核管理的，而是完全由程序控制的，在用户态执行，其主要优势在于它的启动和停止开销非常小，相比于传统的线程来说，可以更有效地实现并发。

协程不是进程也不是线程，而是一个特殊的函数。这个函数可以在某个地方被“挂起”，并且可以重新再挂起处继续运行

一个进程可以有多个线程，一个线程也可以有多个协程运行。在一个线程内可以有多个这样的特殊函数在运行，但是必须明确一点：一个线程中的多个协程的运行是串行的。如果是多核CPU，那多个进程或一个进程内的多个线程是可以并行运行的。但是在一个线程内的多个协程之间运行绝对是串行的，无论CPU有多少核。当一个协程被运行时，其他协程必须被挂起。

进程、线程、协程的对比

  • 协程既不是进程也不是线程，协程仅是一个特殊的函数。协程、进程和线程不是一个维度的。
  • 一个进程可以包含多个线程，一个线程可以包含多个协程。
  • 虽然一个线程内的多个协程可以切换，但是这多个协程是串行执行的，某个时刻只能有一个线程在运行，没法利用CPU的多核能力。
  • 协程与进程一样，也存在上下文切换问题
  • 进程的切换者是操作系统，切换时机是根据操作系统自己的切换策略决定的，用户是无感的，进程的切换内容包括页全局目录、内核栈与硬件上下文，切换内容被保存在内存中，进程切换过程采用的是“从用户态到内核态，再到用户态”，切换效率低。
  • 线程的切换者是操作系统，切换时间是根据操作系统自己的切换策略决定的，用户是无感的。线程的切换内容包括内核栈和硬件上下文。线程切换内容被保存在内核栈中。线程切换采用的是“从用户态到内核态再到用户态”，切换效率中等
  • 协程的切换者是用户，切换时间是用户自己的程序决定的。协程的切换内容是硬件上下文，切换内存被保存在用户自己的变量（用户栈或堆）中。协程的切换过程只有用户态（即没有陷入内核态），因此切换效率高。

问题： 有了协程，还会不会出现线程的切换？

使用协程可以实现轻量级的非抢占式多任务，并在协程之间进行切换，而不需要线程的切换。协程是一种用户级的线程，它在应用程序中定义了显式的切换点，通过协作而不是抢占的方式进行任务切换。

在协程中，任务的切换是由开发者在代码中显式地指定的，而不是由操作系统或调度器自动进行的。这使得协程在某些情况下更加高效和灵活，因为它避免了线程切换的开销和上下文切换的开销。

有协程和出现线程切换没有关系，在线程要出现切换时，该切换就切换，但是线程切换的话也不是随便切换的，是要达到周期才会触发切换的。

协程的弊端

  • 一旦一个协程阻塞的话，那么这个协程所在的线程也会阻塞，一旦阻塞改线程代码就不能向下继续执行了。
  • 协程必须主动让出，这样才能轮到改线程中另一个协程运行。

能否让协程自由的在不同线程中移动，这样就不会因为协程阻塞了某一个线程而导致该线程中其他协程得不到执行？

在此Go语言就对Coroutine做了非常多的优化，从而就推出了Goroutine。

GMP模型

在Go语言中的协程，非常重要的就是协程调度器scheduler和网络轮询器netpoller。

Go协程调度中，有三个重要的角色：

  • M：Machine Thread，对系统线程抽象、封装。所有代码最终都会在系统线程上运行，协程最终也是代码，也不例外。
  • G：Goroutine，Go协程。主要存储了协程的执行栈信息、状态和任务函数等。初始栈大小约为2~4k，其实理论上开启百万个Goroutine不是很大的问题。
  • P：在Go1.1版本就引入了，Processor，虚拟处理器
    • 这里我们也是可以通过环境变量GOMAXPROCS或runtime.GOMAXPROCS()来设置，默认不设置的话就是CPU核心数。
    • P的数量决定着最大可以并行的G的数量。
    • P是有自己的队列的（长度为256），里面放着等待执行的G。
    • P是必须和M绑定在一起的，这样的话P队列里面的G才可以真正的在线程上执行了。

  1. 使用go func创建一个Goroutine g1
  2. 假设当前的P为p1，将g1加入当前P的本地队列LRQ（Local Run Queue）。如果LRQ满了的话就加入到GRQ（Global Run Queue）中。
  3. P中的G要运行就需要找个M绑定一下，p1和m1绑定，m1首先尝试去p1的LRQ中去请求G。如果没有的话就从GRQ中去请求G。再没有的话就，随机从别的P中的LRQ中偷（work stealing）取一部分G到本地的LRQ中。
  4. 假设m1最终是拿到了g1
  5. 开始执行，就让g1的代码在m1线程上运行。
    1. 当g1正常执行完（函数调用完成了），g1和m1解绑，执行第3步的获取下一个可执行的g。
    2. 当g1中代码主动让出控制权，这时g1和m1解绑，将g1加入到GRQ中，在执行第3步获取下一个可执行的g。这种方式很少使用
    3. g1中使用了channel、互斥锁等操作进入了阻塞态（这里的阻塞是协程进入了阻塞态），g1和m1解绑，这了的g1就会放在一边，不会放在GRQ中的，是悬浮态的。在执行第3步的获取下一个可执行的g。如果在阻塞态的g1被其他协程g唤醒后，就会尝试加入到唤醒者的LRQ中，如果LRQ满了，就连同g和LRQ中一半转移到GRQ中。
    4. 出现系统调用
      1. 同步系统调用，这时遇到了同步阻塞系统调用的话，g1会阻塞，m1也会被阻塞，这时m1就和p1解绑。
        1. 但是p1中还有没执行完的G，这时就会去休眠线程队列中获取一个空闲的线程来跟p1绑定，并去p1中队列获取下一个可执行的G来执行；如果这时休眠线程队列中无空闲的线程，就会创建一个新的线程提供给p1来绑定去运行G。
        2. 如果这时m1阻塞结束，这时m1中的g1想要继续运行的话就需要找一个空闲的P去绑定，不过它会优先和原来的p1去绑定。如果没有空闲的P的话就会将g1放到GRQ中，m1就会加入到休眠线程队列中。
      2. 异步网络IO调用，网络IO代码会在GO的底层代码中变成了非阻塞IO，这样就可以使用IO多路复用了。
        1. 当m1在执行g1，执行过程中发生了非阻塞IO的调用（读/写）时，g1就会和m1解绑，这时的g1就会被网络轮询器Netpoller去接管；m1就会在从p1中的LRQ中获取下一个可执行的G来执行，如果LRQ中没有就去GRQ中去拿，实在没有的话就会去其他的P中偷取一些过来给m1来运行。要注意这里的m1和p1没有解绑。
        2. 当g1等待完IO就绪后，g1就从网络轮询器中移回到P1中的LRQ中，如果p1中的LRQ满了的话就会放到GRQ中，g1会重新进入可执行的状态。
        3. 这里我们可以理解网络轮询器Netpoller在内部使用了IO多路复用和非阻塞IO的方式，类似于我们在IO模型中的IO多路复用那代码中的select的循环。在GO中对不用的操作系统MAC（kqueue）、Linux（epoll）、Windows（iocp）提供了这些支持。

异步网络io调用如下图：

网络IO代码会被Go在底层变成非阻塞IO，这样就可以使用IO多路复用了。

m1执行g1，执行过程中发生了非阻塞IO调用（读/写）时，g1和m1解绑，g1会被网络轮询器Netpoller接手。m1再从p1的LRQ中获取下一个Goroutine g2执行。注意，m1和p1不解绑。

g1等待的IO就绪后，g1从网络轮询器移回P的LRQ（本地运行队列）或全局GRQ中，重新进入可执行状态。

就大致相当于网络轮询器Netpoller内部就是使用了IO多路复用和非阻塞IO，类似我们课件代码中的select的循环。GO对不同操作系统MAC（kqueue）、Linux（epoll）、Windows（iocp）提供了支持。

问题： 如果GOMAXPROCS设置为1，这会说明什么？

它指定了GO中Goroutine的并发执行的最大数量为1，这样的话会导致程序的并行运行受限，这样设置会只有一个处理器核心用于执行Goroutine，这样的话就会导致程序在处理并发的任务时就会变得很慢，这样也就没法充分的去利用多核处理器的优势；也就是只有一个线程在运行程序。

为什么会出现这样的情况呢？

因为会启动协程来运行add函数，那么也就是说go add(5, 10)这一句没有必要等到函数返回才结束，所以程序就执行了下一行打印了”2: main end”。这时的main发现没有事情可做了，Go程序启动时也创建了一个协程，main函数就是运行在其中的，我们可以认为是main Goroutine（主协程）。要注意主协程一旦执行结束，也就是进程也结束了，它不会等到还没执行完的其他协程执行完的。
