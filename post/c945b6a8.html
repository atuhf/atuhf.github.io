Channel通道

通道的核心数据结构

channel是一种类型，它可以用来在协程之间传递数据。go语言提倡使用通信来代替共享内存。当一个资源需要在goroutine之间共享时，通道在goroutine之间架起了一个管道，并提供了确保同步交换数据的机制。

在声明通道时，需要指定要被共享的数据的类型。可以通过通道共享内置类型、命名类型、结构类型和引用类型的值或指针。

Go语言中的通道在任何时候，同时只能有一个goroutine访问通到进行发送和接受数据。

在Channel底层是一个先进先出的一个环形队列（固定大小是以环形数组实现的）

  • full或empty都会阻塞
  • send发送
  • recv接收
  • sendx表示最后一次插入元素的index
  • recvx表示最后一次接收元素的index
  • 发送、接收的操作符号都是-

nil通道

nil通道：chan零值是nil，也是可以理解未被初始化通道这个容器。nil通道可以认为只要我们去操作就会阻塞当前协程的容器。这种通道不要创建及使用，阻塞后是无法解除的，在底层源码中写明了无法解除。

非缓冲通道

非缓冲通道是指的容量为0的通道，与上面说的nil通道还是有区别的，我们也叫它为同步通道。这种通道发送第一个元素时，如果没有接收的操作的话就会立即阻塞，直到有接收的操作为止。同样接收时也是如果没有数据发送也就会阻塞在那，直到有数据发送。

缓冲通道

缓存通道就是容量不为0的通道，当通道满了的话再往通道发送数据的话会被阻塞；通道为空时，从该通道接收数据的操作也会阻塞。

单向通道

  • <- chan type 这种定义表示只从一个channel里面拿，说明这是只读的
  • chan <- type 这种定义表示只从一个channel里面写，说明这是只写的

通道关闭

  • 我们可以是通过close(c)来关闭一个通道的
  • 只有发送方才能关闭通道，一旦通道关闭，发送者就不可以在往该通道里面发送数据了，否则会panic: send on closed channel
  • 通道关闭的作用就是告诉接收者后续再无新的数据可以送达了
  • 通道关闭
    • 我们可以通过 data, ok := <-c 或 data := <-c 来从通道中获取数据
    • 正在阻塞等待通道中的数据的接收者，由于通道被关闭，接收者将不会再阻塞，获取数据失败时ok为false，这时返回的值是零值。
    • 接收者依然可以访问关闭的通道是不阻塞的
      • 如果通道内还有剩余的数据，ok为true，接收数据
      • 如果通道内剩余的数据被拿完了，继续接收不会阻塞，ok为false，返回的值为零值
  • 已经关闭的通道，若再次关闭的话会报panic，因此不要重复关闭

通道遍历

1、nil通道

发送、接收、遍历都会阻塞。

2、非缓冲、未关闭的通道

相当于一个无限的通道，迭代不完，会阻塞到下一个元素的到达。

3、非缓冲、关闭的通道

当通道被关闭后，通道就不能再进入新的元素了，那么相当于遍历有限个元素容器，遍历完就结束了。

4、缓冲、为关闭的通道

相当于一个无限的通道，迭代不完，会阻塞到下一个元素到达。

5、缓冲、关闭的通道

当通道被关闭后，通道就不能再进入新的元素了，那么相当于遍历有限个元素容器，遍历完就结束了。

通道死锁

channel满了的话，就会阻塞写操作；channel空了的话，会阻塞读操作。容量为0的通道可以理解为0个元素就满了。

阻塞了当前协程之后会让出CPU，去执行其他的协程，希望其他的协程可以为当前协程解除阻塞。

但是要注意main函数结束了，那就说明整个进程也跟着结束了。

如果在main协程中，执行语句阻塞是，没有其他的子协程可以执行，只剩主协程自己的话，就会自己将自己kill掉，这时就会报一个 fatal error deadlock

如果通道阻塞不在main协程中发生，而是发生在子协程中，这时子协程会继续阻塞着，这会也有可能发生死锁了。但是由于至少main协程是还在执行的，还是值得等待的希望，这里编译器不能帮你识别出死锁。如果真的没有任何的协程帮助改阻塞的协程解除阻塞状态的话，其实上该阻塞的协程解锁无望的话，是已经进入死锁的状态了。

注意：死锁的危害就是可能导致进程还是活着的，但实际上某些协程又未真正的工作而是阻塞，所以我们要有良好的编码习惯，来减少死锁的出现。

struct{}型通道

如果一个结构体类型就是struct{} ，说明该结构体的实例没有数据成员，也就是实例在内存中占用为0

这种类型没有数据构成的通道，非常的节约内存，作用也就是仅仅为了传递一个信号的标志。

通道并发

GO语言是采用了并发同步模型叫做 Communication Sequential Process 通讯顺序进程，这是一种消息传递的模型，在Goroutine间传递消息，而不是对数据进行加锁来实现同步访问。在Goroutine之间通过channel来同步和传递数据。

  • 多个协程之间通讯的管道
  • 一端写入数据，一端拿走数据
  • 同一时间，只有一个协程可以访问通道的数据
  • 协调协程的执行顺序

如果多个线程都使用了同一个数据，这时就会出现竞争的情况。因为线程的切换是不会听从程序员的意志，时间片用完了就会切换。往往解决的办法就是加锁，让其他线程不能对共享数据进行修改，从而保证逻辑正确。但是锁的引入就会影响并行的效率。

协程泄漏

主要原因

  • 协程阻塞，未能如期结束，之后就会大量累积
  • 协程阻塞最常见的原因就是跟通道有关
  • 由于每个协程都要占用内存，所以协程泄露也会导致内存的泄露

因此，如果你不知道你创建的协程何时能够结束，就不要使用它，否则的话就可能会引起协程的泄露，从而会导致内存的泄露。
